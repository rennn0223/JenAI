# JenAI v0.25.0 - Safety hardening and OpenAI SDK alignment

This release closes six safety and reliability gaps found during a full architecture review.

## Highlights

- Added a single `NavigationGateway` used by CLI, TUI, WebUI, MCP, daemon, missions, patrols, and agent tools. Twin Gate and watchdog policy can no longer be bypassed by a direct route execution call.
- Emergency halt now commands zero velocity before the potentially slow Nav2 cancel-all round trip, then pulses zero again afterward.
- Odom depth avoidance fails closed when frames are missing or stale and reports `sensor_unavailable`; corridor checks now follow the actual target heading.
- Twin Gate G4 returns `refer` when the simulated final pose is unavailable instead of treating an unverified endpoint as a pass.
- Configuration writes preserve nested safety profiles through TOML round trips and use atomic replacement. Unsafe negative/non-finite limits and invalid profile relationships are rejected at load time.
- Official OpenAI agent profiles now use the Responses model. OpenAI-compatible custom endpoints, NVIDIA NIM, and Ollama continue through Chat Completions.
- Approval interruptions use the Agents SDK native `RunState.to_json/from_json` format and can be restored after a TUI restart. State files are atomic, mode `0600`, exclude tracing API keys, and are claimed once to prevent replay.
- Removed the unused direct LiteLLM dependency.

## Compatibility

- Existing OpenAI-compatible provider profiles continue to work unchanged.
- `[avoidance] enabled = true` is intentionally stricter: loss of fresh depth data stops the odom driver instead of degrading to blind motion.
- New optional setting: `depth_timeout_s = 1.0` under `[avoidance]`.
