## 修正一:`/run` 批准後工具不執行(最高優先 known issue)

- **Root cause(黑紙白字)**:openai-agents SDK 的 `resolve_resumed_context` 在 resume 收到 `context=` 參數時,會**建一個全新的空 wrapper 蓋掉 `run_state._context`**——而 `state.approve()` 記錄的批准就存在被蓋掉的 wrapper 裡。JenAI 的 `resume_with_approvals` 恰好傳了 `context=ctx`,批准全數蒸發 → 工具重新 interrupt(audit 實測 28ms)→ 被迴圈偵測器誤判成「模型迴圈」
- **修法**:resume 不再傳 `context=`——in-memory 路徑 state 在暫停時已抓著同一個 ctx,跨重啟路徑由 `take_pending_state` 的 `context_override` 注入
- **回歸測試(真 SDK,scripted model,無網路)**:批准 → 工具本體必須執行 → run 完成;rejection → 工具不得執行、模型收到拒絕訊息。**雙向驗證:舊碼兩測皆紅,新碼全綠**——rejection 路徑原來也一直是壞的,一併修復

## 修正二:壞 action 默默導航到原點(fail-closed 違規)

- 實測抓到:35b 把 preview 回應整包再包一層 `{"outgoing_action": …}` 餵給執行工具,`goal.get("pose") or {}` 讓目標默默變成 **(0, 0)——車真的開到地圖原點並回報 succeeded**
- **修法(單一出口安全地板)**:`navigate_with_fallback` 入口驗證 `goal.pose.x/y` 必須是有限數字(bool 不算),壞結構一律 `failed` +「nothing was sent」,所有入口(TUI/MCP/daemon/agent)受惠;agent 工具層容錯解包模型的包裝、`route_execute_tool` 的成敗如實回報(不再寫死 ok=True)
